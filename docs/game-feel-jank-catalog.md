# ゲームフィール不整合カタログ

プレイヤーが「なんかおかしい」とだけ言語化する違和感の正体は、ほぼ常に**入力とフィードバックの因果が壊れている瞬間**である。コトダマ・リフォージv1のフィール監査([詳細全文](game-feel-audit.md)、確認済み17件)から帰納した、タッチUIゲーム全般のチェックリスト。

発端はプロダクトオーナーの一言「回答をタッチした瞬間タイマー止まるとかなんかおかしい」——この一文の裏に、ゴーストクリックによる誤回答(実害バグ)・偽タイマー(メタファー齟齬)・入力凍結(テンポ破壊)の3クラスが隠れていた。**ユーザーの体感報告は症状名であって診断名ではない**。

## クラス1: 入力の因果が壊れる(最優先で潰す)

- **ゴーストクリック**: pointerupでDOMを差し替えると、ブラウザが同じタップに合成するclickが「同座標に出現した新要素」に着弾する。タッチ実機のみ発生し、マウステストでは絶対に再現しない。→ **画面差し替えごとにタイムスタンプを記録し、~300ms以内の入力を捨てる共通ガード**で、ダブルタップ貫通・すり替わり誤タップも一括解決できる。
- **ジェスチャーのフェーズ跨ぎ**: スワイプ中に自動遷移が起きると、離した瞬間の画面に対して解釈され「見てもいない選択肢への盲目回答」になる。→ pointerdown時のフェーズを記録し、変わっていたら破棄。
- **指のロール**: ボタン上でタッチ開始→30px滑って離す、はモバイルの通常タップ。方向フリックと競合させると「押したのと違うボタン」が発動する。→ ボタン上で始まったジェスチャーはそのボタンを優先。フリックは背景開始+60px以上に限定。
- **死にゾーン**: ブラウザのタップスロップ(~10px)超過でclickが合成されない入力を、自前のpointerupが拾わないと「押したのに無反応」になる。

## クラス2: タイマー・ゲージの嘘

- **偽タイマー**: 「減っていくバー」はプレイヤーには必ず制限時間と読まれる。実際は演出ペーシングで、触ると中途半端な位置で消える——これが「タイマーが止まる」体感の正体。**見えるタイマーは必ず何かを賭けさせ、効いているタイマーは必ず見せる**(逆転が最悪)。
- **数字とゲージの不一致**: 同じ量を数字とバーで二重表示するなら、すべての更新経路で両方を同期する。片方だけ更新する経路が1本でもあると、いつか食い違った画面をプレイヤーが見る。
- **CSS transition頼みの開始**: innerHTML直後の単一rAFでtransitionを開始すると、フレーム位相次第で初期値が描画されず即終端値になる。Web Animations API(`el.animate`)は位相に依存しない。

## クラス3: 凍結とテンポ税

- **入力ロックアウト**: 「演出のための自動待ち」をスキップ不能にすると、せっかちなプレイヤーは毎回「フリーズ」を体感する(v1ではランの42%が入力無効時間だった)。→ **自動進行は無操作時のフォールバックにし、タップで常に先送りできる**ようにする(誤タップ防止に先頭250msだけガード)。
- **演出の空中分解**: 演出の長さ>画面の寿命だと、毎回途中で切断される(+ポイントが85%地点で消える等)。演出は遷移より必ず短くするか、遷移を跨ぐオーバーレイ層に置く。
- **計測してから直す**: 「中断回数 ÷ 報酬回数」「強制待ち時間 ÷ 総プレイ時間」をプレイログから実測すると、体感問題が定量化できる(v1: 罰17:報酬1、待ち42%)。

## 運用上の教訓

- マウスのヘッドレステストはタッチ固有バグ(ゴーストクリック)を**構造的に検出できない**。`hasTouch: true` + `touchscreen.tap()` の回帰テストを必ず別に持つ。
- 関連: 学習ゲームの搾取パターン(インセンティブの整合、初回実装レビューで整理)に対し、本カタログは**知覚の整合**を扱う。両方揃って初めて「ちゃんとしたゲーム」になる。
