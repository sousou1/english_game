`docs/nakige-emotional-design.md`(感動設計)の上位に立つ「どう書き・どう通すか」の手順書。
以後のストーリー(章・イベント)執筆は、本ワークフローに従う。

## 0. 不変の鉄則(プロセスを変えても死守)

`docs/scenario-structure.md` §7・§8 の鉄則は全フェーズで維持する:
- **誤答は物語を変えない。選択だけが物語を変える。** フラグは加算のみ・負効果ゼロ。
- **説教禁止 / 裏テーマ(沈黙=声に出せない恐怖)は言語化しない。**
- **詠唱語は必ず `data/words.js` 実在語**(`w` キーで検証)。固有名詞は対象外。
- 1シーン2〜8行。8行上限は絶対。世界観・システムは現行のまま変えない。

## 1. 執筆フェーズ(演出・イラスト概要込みで書く)

ストーリーは**プレーンな台詞だけでなく、演出とシーンイラストの概要を含めて**執筆する。
各シーンに次を付す(`docs/drafts/` の素案フォーマット):

| 要素 | 内容 |
|---|---|
| **本文** | 2〜8行の地の文/台詞(既存の話者「…」形式。立ち絵検出と互換) |
| **演出** | BGMの気分(幸福の音/不穏/喪失)・SE・立ち絵と表情・カットイン・画面効果・間。`earned-catharsis §1`の振幅(緩→急の落差)を演出で設計する |
| **シーンイラスト概要** | そのシーンの挿絵ブリーフ。`構図/被写体/時間帯・光/画風/整合タグ`。整合タグ=後段の画像整合の鍵(例: `村=中世風・木造・茅葺`/`王都下町=石造・雑多`)。全シーンに付けず、節目シーンに付す(予算は`docs/scenario-structure.md`§11準拠) |
| **学習接続** | そのシーンが入力モード(詠唱/お使い/移動)で進むなら、使用語(実在検証済み)と演目種別 |

章単位の起伏は `docs/nakige-emotional-design.md` のテンプレ(開幕の緩を厚く→小さな喪失の分散
→モチーフの意味変容回帰→完全勝利にしないほろ苦)を踏襲する。

## 2. レビュー・ゲート(マージ前に必ず通す)= 多ペルソナ独立レビュー方式【必須・標準手順】

素案・確定稿・章の書き直しは、**例外なくこの方式を通す**(2026-06-13 ユーザー承認。「このレビュー方式を今後も必ず使う」)。
要点は **(a) 複数の異なる視点(ペルソナ)を、(b) それぞれ独立したサブエージェントで並行に走らせ、(c) high指摘がゼロになるまで反復する** こと。1エージェントに兼任させない(視点が混ざり甘くなる)。

### 2.1 手順(回す順番)
1. **並行ファンアウト**: 下記3関門を**独立サブエージェント3体に同時起動**(`Agent` を1メッセージで複数呼ぶ)。各エージェントには評価軸の正典(下表の必読)を渡し、忖度なしで具体的シーンID/行を引用させる。
2. **集約と反映**: high指摘(=「直さないと出せない」)と、ペルソナの最重要指摘を素案に反映する。
3. **同一エージェントへ再提出**: 修正後、**同じレビューアに `SendMessage` で差分を見せて再判定**させる(新規起動より、前回指摘との整合が取れる)。
4. **収束まで反復**: 全関門が「出せる/合格」かつ **high指摘ゼロ** になるまで 2〜3 を繰り返す。1関門でも「差し戻し」なら未収束。

### 2.2 3つの関門(=最低限の視点。題材に応じてペルソナを足してよい)
1. **編集者レビュー**: 掴み(最初の数分)・ペーシング・キャラ声の一貫性(§8-9)・伏線の設置と回収・
   振幅(`earned-catharsis 5機構`)・テンプレ疲れ回避。**10点満点の点数**と「直さないと出せない(high)」指摘を出す。
2. **顧客ペルソナ指摘**: ターゲット(中2男子ほか。`docs/playtest-persona-2026-06-13.md`の人物像)に**なりきって**通読し、
   「テスト臭」「説教臭」「ダサい」「飛ばした(▶連打)箇所」を率直に。良かった点も挙げさせる(過剰な差し戻し防止)。
   ※ターゲット層が複数なら**ペルソナを複数体に増やす**(例: 英語得意な層/物語重視の層)。これが「いろんなペルソナ」の核。
3. **entertainment-vault 合格判定**: vaultの`earned-catharsis`(成立条件/壊れる条件)・`nakige`(尺不足=愛着不足等)・
   学習ゲーム教訓(`lessons/`)に照らし、**壊れる条件に該当しないこと**を判定。該当すれば差し戻し。

### 2.3 合格基準
- 3関門すべてで「出せる/合格」。**high指摘ゼロ**。
- 詠唱語・供給語の**実在チェック(`data/words.js`)がグリーン**(`w`キー検証・重複/衝突ゼロを機械検証)。
- 機械検証(実在/重複/ゲート8割ルール)は**レビュー前に済ませ**、関門には設計・物語の質に集中させる。

## 3. 素案のマージ/公開(ユーザー閲覧)

合格した素案は `docs/drafts/<章slug>.md` として **github にマージ/push** する。ユーザーがリモートで閲覧する。
ここで一旦**停止し、ユーザーの go を待つ**(画像生成はユーザーの go の後)。

---

## 4. 画像生成(ユーザーの go の後)

`anima-illust` スキル(ローカル Cosmos-Predict2 / ComfyUI、`~/llm/image_gen/gen.sh`)で挿絵を生成する。

- **シーン間の整合性を最優先**: §1の整合タグで世界の見た目を統一する(例: 中世風の民家なら様式・素材・色調を全シーンで揃える)。
  キャラの容姿・服・髪飾り、土地の建築様式、時間帯の光を破綻させない。
- **claude vision でレビュー**: 生成画像を vision で読み、整合タグからの逸脱(様式バラつき・キャラ崩れ)を検出して再生成する。
- 立ち絵・表情は既存規約(`ui.js` PORTRAITS / moodOf)と互換に。**絵の最終可否は人間レビュー**(AIビジョンは整合チェック補助に留め、採否判定はしない。handoff規約準拠)。

---

## 5. スクロール閲覧ページの作成(指摘しやすさ最優先)

生成画像を組み込んだ**HTMLスクロールページ**を作る。要件:

- 画像を埋め込み、上から読めるスクロール形式。本文・演出ノート・挿絵を流し込む。
- **章の名前を明記**(見出しに章タイトル)。
- **各画像に id を振る**(例: `img-c02-010`)。ユーザーが「この id の絵が違和感」と指摘しやすくする。
- 本文ブロックにも参照しやすい見出し/シーンIDを付す。

ユーザーがこのページを見て、**違和感のある画像・文章を指摘**する。

---

## 6. ゲームへの組込み

ユーザー指摘を反映後、確定稿を `js/scenario.js`(章シーン)・`data/events.js`(イベント)・
立ち絵/挿絵アセットとしてゲームに組み込む。組込み後は通常の検証(全テスト・JSパース・
`tests/_shot_*.mjs`実機撮影・sw版上げ・push)を通す。
