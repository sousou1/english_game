# 設計: 単語マスター(昇華)機能 ―『ともしび』D1

> 出典: プレイテスト feedback(2026-06-17)「呪文書で各単語の回答数が30になると『昇華』ボタン→押すとローテ除外＋基礎能力↑。昇華数に応じて新イベント(新語)を増やし、常に一定数の単語でローテ」。
> 位置づけ: backlog D1。**Tier3=実装前にこの設計をユーザ確認**(経済不変インバリアントを意図的に変えるため)。D2(昇華数で解禁)の土台＝先に本機能で「昇華数」を定義する。

## 1. 目的
- 覚えた語が増えるほど出題ローテが膨らみ「簡単な既習語ばかり当たる」問題を解消。
- 熟達した語を**卒業(昇華)**させてローテから外し、代わりに**恒久的な基礎能力ボーナス**に変換。
- 昇華数を進行通貨にして、新イベント(新語供給)とメインストーリー(D2)を解禁→**常に“ちょうどよい難度の語”でローテ**。

## 2. 現状データ(実装済の足場)
- `card = {S, D, last, due, reps, lapses}`(`js/srs.js`)。**reps=SRS回答回数**。別に `p.taps[w]`=**バトル詠唱回数**(呪文書詳細に「唱えた回数」で表示)。
- 基礎能力(タップ価値)= `p.vref`(`js/pool.js:260` 移動平均)。恒久倍率 `globalMult() = collMult(学習語数) × 2^ring × armoryMult`(`pool.js:85`)。`collMult` は学習語数 25/50/100/200/300/500/750 で ×2(`pool.js:40`)。
- 新語供給はイベント限定(`data/events.js` gate.read)。学習語数 = `learnedEntries()`(reps>0 のカード)。
- 進行ゲート = settled語数(S≥3) + costW(ゴールド)。

## 3. 昇華の定義(提案)
### 3.1 条件
- **語ごとの「回答数」が閾値に達したら昇華可能**。
  - 提案: **count = card.reps(SRS) + (p.taps[w] 由来の運用回数)** を統合した「習熟回数」を新フィールド `p.mastery[w].ans` で持つ。閾値 **30**(ユーザ指定)。
  - ⚠️決定要: 「回答数」を (a) SRS reps のみ / (b) バトル taps のみ / (c) 両者合算。呪文書に出ているのは taps なので体感は(b)寄り。**既定案: (c)合算**(学習とプレイ両方を反映)。
- かつ **S≥ある閾値(例: 黄金 S≥30 相当)** を併用するか? ⚠決定要(回答数だけだと未定着でも昇華できてしまう)。**既定案: 回数30 かつ S≥7(銀)** を満たすと昇華可能。

### 3.2 効果(押下時)
1. **ローテから卒業(ユーザ確定 2026-06-18)**: 昇華語は通常ローテから外すが、**完全除去ではなく10%の低確率で復習に再出題**。再出題抽選は**過去に間違えた回数(lapses)が多い語ほど選ばれやすい**重み付け(`weight ∝ lapses+1`)で、誤りやすい語を忘れさせない。実装: 出題プール構築時に確率10%で「昇華語プール」から重み付き抽選して1語混ぜる(`pool.js`/`workshop.wakeQueue`)。残り90%は通常(未昇華)ローテ。
2. **基礎能力↑**: `p.mastery.count`(昇華総数)に応じた**恒久倍率「昇華係数」** を `globalMult()` に乗算で追加。
   - 提案式: `sublMult = 1 + SUBL_K × 昇華数`(線形・微増) または マイルストーン式(collMult同様)。⚠上げ幅は経済シムで決める(§6)。
   - **collMult は据え置き**: 昇華語も「学習済み(コレクション)」には数え続け `collMult` の学習語数は減らさない(=既存の不変条件を壊さない)。除外するのは“出題ローテ”だけ。
3. 演出: 昇華した語は呪文書で金色「昇華済」表示。通常ローテからは外れるが、10%枠で時おり復習に顔を出す(§3.2-1)。

### 3.3 ローテ語数の維持
- **アクティブ・ローテ = 学習済 − 昇華済**。目標サイズ(例 **20語前後**)を保つよう、**昇華数が一定進むごとに新イベントを解放**(新語8語/本)。
- 接続: 昇華が進む→新イベント解禁(§5/D2)→新語がローテ補充→ローテサイズが一定帯に収束。
- ⚠決定要: 目標ローテサイズ(20? 25?)と、昇華何個ごとに1イベント解放するか(例: 昇華4個ごと=新語8語、で純増+4/帯維持)。

## 4. データ/セーブ(schema v6→v7 migrate)
- 追加: `p.mastery = { count: <昇華総数>, words: { <w>: { ans, subAt } } }`。`ans`=習熟回数、`subAt`=昇華時刻(null=未昇華)。
- migrate: 既存 `p.cards`/`p.taps` から `ans` を初期化(reps+taps)。`count=0`、既存語は未昇華。`SCHEMA_VERSION` を 6→7、`storage.js migrate` に追記(後方互換)。

## 5. UI
- 呪文書(`renderSpellbookSheet`): 各語に習熟回数 `ans/30` の小バー。閾値到達かつS条件OKで **「昇華」ボタン**を行内に表示。押下で確認→昇華。
- 昇華済語: 金色タグ「昇華済」・出題対象外を明示。
- ステータス or 呪文書ヘッダに **昇華数** と **次のイベント/メイン解禁までの残り昇華数**(D2)を明示(feedback要望)。

## 6. 経済不変インバリアントの扱い(重要)
- **守る**: `IDLE_K=0.21`、`collMult` マイルストーン(25/50/.../750→×2)、`TIER_MULT`、`BATTLE.chapterBosses`、`enemyHp` 式 ―― `tests/invariants.test.js` で凍結中。**これらは変えない**。
- **意図的に足す**: `globalMult()` に **昇華係数 `sublMult`** を新規乗算(既存因子は不変)。これは新しい上昇源なので、**30日シム(`tests/_sim_v6.mjs`)を再実行して**「放置≫タップ」事故やゲート到達日の破綻が無いことを確認してから係数確定。
- インバリアント追加: 「昇華係数は昇華数0で1.0(無昇華時は現行と完全一致)」を test 化(回帰防止)。
- ⚠ `SUBL_K`(上げ幅)は**シム結果で決める数値**＝設計段階では未確定。保守的(微増)から入れて再シムで上げる。

## 7. 影響ファイル
- `js/srs.js` or 新規 `js/mastery.js`: 習熟回数の加算・昇華可否判定。
- `js/storage.js`: schema v7・`p.mastery`・migrate。
- `js/pool.js`: `globalMult` に `sublMult`、出題タイルから昇華語を除外、`learnedEntries`(collMultは昇華語も含む/ローテは除外)を分離。
- `js/workshop.js`: `wakeQueue`(復習)から昇華語を除外。
- `js/ui.js`: 呪文書の昇華ボタン・習熟バー・昇華数表示。
- `data/events.js` / 解禁ロジック: 昇華数による新イベント解放(D2と統合)。
- `tests/invariants.test.js` + `tests/_sim_v6.mjs`: 凍結値据え置きの確認＋昇華係数の再シム。

## 9. 決定(ペルソナ4体評価で確定 2026-06-18)
> 中2学習者 / 学習科学(SRS) / ディレクター(So) / 継続・経済バランス の4レンズで評価し合意点を採用(ユーザ指示「ペルソナ作って評価して」)。数値は全て30日シム(`_sim_v6.mjs`)待ち＝ここでは“形と基準”を確定。

- **#1 回答数の定義** → **合算 `ans = reps + min(その日のtaps, 日次キャップ)`**(合算3/4)。学習科学の「連打量産で“覚えた”の代理が崩れる」懸念を **tapsの日次キャップ(例 +10/語/日)** で緩和。閾値30。呪文書バーは合算1本「習熟 ans/30」(内訳はdetail)。1アクション=1加算。migrate初期ansは `min(reps+taps, 25)` でクリップ(既存ユーザの即昇華ラッシュ防止)。
- **#2 未定着昇華の防止** → **回数30 かつ S≥7(銀)併用(併用は全員一致)**。S閾値は7既定、**シムで「回数30到達 vs S到達」の律速を見て7〜10に調整**(学習科学はtier境界S≥10を推奨)。UI: 満タンでもS不足は**鍵付き非活性ボタン＋理由一行**、回数/定着の2点を可視化。
- **#3 ローテ卒業の範囲** → **ユーザ確定=10%低確率・lapses重み再出題**(§3.2-1)。修行(wakeQueue)では昇華語を🔥(復習どき)として出さず、再出題はバトル/復習プールの10%枠のみ(卒業の見た目を壊さない)。
- **#4 基礎能力↑の形** → **マイルストーン式(collMult同型・到達ごと×)で確定(全員一致)**。線形は経済原則I1(連続指数はコスト側のみ/生産側は離散×2)に反し発散方向のため却下。段の谷間対策に**即時一時金(昇華ボーナス)**を併設。`sublMult(0)=1.0` をinvariant化。
- **#5 ローテ目標＋解禁頻度** → **アクティブ20語(帯14〜26)・昇華4個ごとに新イベント(新語8語)→純増+4で自己収束(全員一致)**。序盤は学習語が最初のcollMult節目(25語)に届くまで**昇華UIを伏せる**。解禁直後の新語8語は**分割投入**(▶連打/間延び回避・B2の1語ずつclozeと相互依存)。
- **#6 昇華の取消** → **クールダウン付き取消(同一セッション/24h以内のみ「戻す」可、以降恒久)＋確認ダイアログ必須(3/4)**。取消時はansを閾値直下(29)に戻し即再昇華不可(係数の出し入れ悪用防止)。解禁済みイベントは据え置き・係数だけ巻き戻す。

### 9.1 横断事項(実装前提)
- **単一進行通貨**: 昇華数を #4ノード/#5解禁/D2解禁で同じ刻みに揃え、呪文書ヘッダに〔昇華数 / 次解禁まであとN〕の1ペアを出す(feedback「次解禁までの昇華数を明示」を満たす)。
- **数値はシム待ち**: SUBL_K(段倍率)・S閾値・段間隔・解禁ピッチは `_sim_v6.mjs` 30日再シムで「放置:タップ比0.3〜3.0」「ローテ帯14〜26維持」を同時に満たすか検証して確定。B2(1語ずつcloze)で供給レートが変わるため**新供給前提で回し直す**。
- ✅ **IDLE_K表記揺れ=誤検知**: `0.21`/分 = `0.0035`/秒×60(economy.js は CpM、economy-spec は CpS の同値)。別係数ではない。
- **B2連動**: 新語は1語ずつcloze(複数語ビート全廃)を前提にローテ供給を順次化。B2はD1ローテの土台なので **B2→D1 の順**が手戻り最小。
