# 設計: 進行モデル変更(昇華数で解禁・costW廃止・昇華数で新イベント解放) ―『ともしび』D2

> 出典: プレイテスト feedback(2026-06-17)「昇華数によってメインストーリーも解禁していく/解禁ポイントを設計/次のイベント・メイン解禁までの昇華数を明示/今のお金(costW)解禁は不要なので削除」。
> 位置づけ: backlog D2。**Tier3=実装前にこの設計をユーザ確認**(進行/経済の core を変える)。D1(昇華)の上に乗る。

## 1. 目的
- 進行通貨を**ゴールド(costW)→ 昇華数**へ。「金を貯めて解禁」をやめ、**“言葉を極めた数”でメイン/イベントが開く**マスタリ駆動の進行に。
- 昇華で語がローテから卒業する分、**昇華数で新イベント(新語8語)を解放**してローテ語数を一定帯に維持(D1 §3.3 の補充をここで実装)。
- 「次の解禁まであと何昇華」を常時明示(D1の呪文書ヘッダを進行表示に拡張)。

## 2. 現状の解禁機構(変更対象)
- **章ゲート(settled建て)**: `js/scenario.js` の最終シーンに `gate:{settled:N}`。c01_180=15、c02_130=40(arc-plot §4 の段 15/40/90/…/450)。判定 `js/ui.js canPassGate`(settledCount = S≥3 の語数)。
- **解錠費 costW(ゴールド)**: クエスト先頭シーンに `costW`(c01_060=0.3 / c01_110=0.6 / c01_160=0.6 / c02_040=0.3 / c02_080=0.6 / c02_120=0.6)。`storyCommit` が `sceneCost` 分のゴールドを消費。不足は B1 で無効表示化済。
- **kills ゲート**: c01_060 `gate:{kills:2}`(戦闘チュートリアル)。
- **イベント解放**: `data/events.js` の `gate.read:"cNN_xxx"`(該当シーンを読むと解放)。
- インバリアント: `tests/invariants.test.js` のシナリオ整合「章ゲートは arc-plot の段[15,40,…]を昇順」。

## 3. 中心的な論点(設計の肝)
- ⚠ **昇華は“遅い”**: 1語の昇華に ans30(=reps+min(taps,20)) かつ S≥7 が要る(D1)。settled(S≥3)より遥かに深い到達。
  - 例: c02ゲートは settled40 ≒ day6(検証済)。一方「昇華5語」は語を深く根づかせる必要があり、はるかに後。
  - → **メイン章ゲートを“純・昇華数”にすると進行が極端に遅くなる**。かつ序盤は昇華数0で**ブートストラップ問題**(昇華するには語が要る→語はイベント→イベントは昇華数で解禁…の循環)。
- 結論: **ハイブリッド**が現実的。下の §4 で3案を提示し、既定案を置く。

## 4. 新機構の設計(3案・既定案つき)
### 案A(既定): 二層ゲート ―「メイン章=settled据え置き / 解禁の“伸びしろ”と新イベント=昇華数」
- **メイン章ゲート(settled)は据え置き**(arc-plot の段[15,40,…]・経済シム検証済を壊さない)。進行の主軸は従来どおり“学んだ語数”。
- **costW(ゴールド解禁)は廃止** → 代わりに**クエスト解禁を「読む(gate.read相当)」だけに**(無料化)。金は戦闘/装備の通貨として残すが“進行ゲート”からは外す。
- **昇華数は“補充と寄り道”の通貨**: 昇華数マイルストーンで**新イベント(新語8語)を解放**し、ローテを補充(D1 §3.3)。さらに**サブ供給(おつかい)や隠し/周回要素**の解禁にも使う。
- 「次解禁まであと N昇華」は**新イベント解放**の残数として明示。
- 長所: 経済/章ペーシングを壊さず、昇華に“ローテ補充+寄り道解禁”の明確な役割を与える。実装も局所的。短所: ユーザ要望「メインも昇華数で解禁」を**部分的にしか満たさない**。

### 案B: メイン章ゲートを「settled かつ 昇華数」の二条件に
- 章ゲート = `{settled:N, subl:M}` の AND。Mは小さめ(例 ch2=1, ch3=2, ch4=4…)で“最低限の昇華”を要求。
- costW廃止。新イベントは昇華数で解放(案Aと同じ)。
- 長所: メイン進行に昇華が直接効く(ユーザ要望に近い)。短所: 昇華が遅いと settled は満ちても昇華が律速で**待ち**が出る→30日シムで「settled到達 vs 昇華到達」の律速を要検証。Mを保守的にしないと序盤で詰む。

### 案C: メイン章ゲートを“純・昇華数”に置換(ユーザ要望の最大解釈)
- 章ゲート = `{subl:M}` のみ(settled廃止)。costW廃止。
- 長所: 完全にマスタリ駆動。短所: **進行が大幅に遅くなる**＋序盤ブートストラップ必須(ch1は無条件解禁、ch2以降から昇華ゲート)＋経済シム(settled前提)を全面作り直し。リスク最大。

> **おすすめ = 案B**(ユーザ要望「メインも昇華数で解禁」を満たしつつ、settled併用でブートストラップ/詰みを回避)。ただし案Aが最も安全。**どれにするかはユーザ判断**(§9)。

## 9. ユーザ確認したい決定点
1. **採用案**: A(二層・最安全) / **B(settled+昇華AND・おすすめ)** / C(純昇華・最大変更)。← 最重要
2. **costW廃止の範囲**: 進行ゲートから完全撤去で確定か(金は戦闘/装備通貨として存続)。
3. **(案B/Cなら) subl ゲート値**と**新イベント解放ピッチ**(4昇華ごと?)を、30日再シムで詰める方針でよいか。
4. **「次解禁までの昇華数」表示**を物語シート/メインのどこに出すか(呪文書ヘッダは実装済)。
5. **昇華数で新イベント解放**の“解放先”が要る = **C1(第3章以降の配線)に依存**。ch2までは補充先が少ないので、C1と並行 or C1後に実値化。
